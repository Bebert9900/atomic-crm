# Feature : Enregistrement d'appels + Transcription & Analyse IA Gemini

Cette feature ajoute la possibilité d'enregistrer le micro pendant un appel (cold calling en haut-parleur) directement depuis la liste des contacts du CRM, de transcrire automatiquement l'audio via l'API Google Gemini, **puis de générer un résumé de l'appel + des conseils pour les suivis email/SMS**.

Comparaison effectuée vs `https://github.com/marmelab/atomic-crm.git` (branche `main`).

---

## Vue d'ensemble du flux

1. L'utilisateur clique sur l'icône 🎤 sur la ligne d'un contact dans la liste
2. Une modale s'ouvre avec : sélecteur de micro, contrôles record/pause/stop, prévisualisation
3. À la sauvegarde :
   - L'audio (`.webm`) est uploadé dans le bucket Supabase `attachments`
   - Une ligne est créée dans `contact_recordings` (`status: pending`)
   - L'Edge Function `transcribe_recording` est appelée en arrière-plan
4. L'Edge Function effectue **2 appels Gemini successifs** :
   - **Appel 1** : transcription audio → texte (modèle `gemini-2.0-flash`)
   - **Appel 2** : envoi de la transcription pour générer un JSON structuré contenant :
     - `summary` : résumé de l'appel en 3-5 phrases (points clés, besoins, objections, prochaines étapes)
     - `email_advice` : conseil détaillé pour rédiger un email de suivi
     - `sms_advice` : conseil détaillé pour rédiger un SMS de suivi
5. L'enregistrement apparaît dans la sidebar de la page contact avec :
   - 🔊 Lecteur audio
   - 📋 Résumé de l'appel (encadré bleu)
   - ✉️ Conseil pour l'email de suivi (encadré vert)
   - 💬 Conseil pour le SMS de suivi (encadré violet)
   - 📝 Transcription complète (encadré gris)

| Fichier | Rôle |
|---|---|

## Fichiers modifiés

### Schéma base de données

#### `supabase/schemas/01_tables.sql`
Nouvelle table `contact_recordings` :

```sql
create table public.contact_recordings (
    id bigint generated by default as identity primary key,
    contact_id bigint not null,
    storage_path text not null,
    duration_seconds integer,
    transcription text,
    transcription_status text not null default 'pending',
    summary text,
    email_advice text,
    sms_advice text,
    created_at timestamp with time zone not null default now(),
    sales_id bigint
);

create index contact_recordings_contact_id_idx on public.contact_recordings using btree (contact_id);

alter table public.contact_recordings
    add constraint contact_recordings_contact_id_fkey foreign key (contact_id) references public.contacts(id) on update cascade on delete cascade;

alter table public.contact_recordings
    add constraint contact_recordings_sales_id_fkey foreign key (sales_id) references public.sales(id);
```

#### `supabase/schemas/04_triggers.sql`
Trigger pour auto-renseigner `sales_id` à l'insertion :

```sql
create or replace trigger set_contact_recordings_sales_id_trigger
    before insert on public.contact_recordings
    for each row execute function public.set_sales_id_default();
```

## Commandes pour faire fonctionner

### 1. Appliquer le nouveau schéma de DB

La table `contact_recordings` contient les colonnes suivantes (à inclure dans la migration) : `transcription`, `transcription_status`, **`summary`**, **`email_advice`**, **`sms_advice`**.

```bash
# Génère un fichier de migration depuis les changements de schéma
npx supabase db diff --local -f add_contact_recordings

# Applique la migration
npx supabase migration up --local
```

Alternative rapide (⚠️ destructif, supprime les données locales) :
```bash
npx supabase db reset --local
```

### 2. Démarrer l'Edge Function en local

Dans un terminal séparé :
```bash
npx supabase functions serve transcribe_recording --env-file supabase/functions/.env
```

### 3. Démarrer le CRM

```bash
make start
```

### 4. Déploiement en production

```bash
# Pousser le schéma sur la DB distante
npx supabase db push

# Déployer l'Edge Function
npx supabase functions deploy transcribe_recording

# Configurer la clé Gemini sur le projet distant
npx supabase secrets set GEMINI_API_KEY=your_key_here
```

## Analyse IA — détails techniques

L'Edge Function utilise un **prompt structuré** qui demande à Gemini de répondre en JSON valide :

```
Tu es un assistant commercial expert en cold calling. Voici la transcription d'un appel...
Réponds STRICTEMENT en JSON valide avec cette structure:
{
  "summary": "...",
  "email_advice": "...",
  "sms_advice": "..."
}
```

Le code parse la réponse JSON (avec strip des backticks markdown au cas où Gemini en ajoute), et si le parsing échoue, le texte brut est sauvegardé dans `summary` en fallback.

## Limitations connues / Améliorations possibles

- **Pas de polling automatique** : le statut "Transcription..." → "Transcribed" ne se met à jour qu'au prochain refresh React Query (clic sur l'enregistrement ou rechargement de la page).
- **Pas de bouton de suppression** des enregistrements depuis l'UI.
- **Pas de cleanup automatique du fichier Storage** lorsqu'on supprime une ligne `contact_recordings` (à implémenter via trigger comme pour `contact_notes`).
- **Format audio fixe** : `audio/webm;codecs=opus`. Gemini accepte ce format mais d'autres formats pourraient être supportés.
- **Pas de limite de durée** d'enregistrement côté frontend.
